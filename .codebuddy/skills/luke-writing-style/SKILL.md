# Luke Writing Style Skill

## 目标
把输入素材（草稿、口述、要点、代码片段、案例）转成**有故事感 + 有工程抓手**的中文技术文章，风格贴近 `ai-思考/` 现有内容。

## 何时触发
出现以下任一信号就触发：

- 用户要写/改/润色技术文章（尤其中文长文）
- 用户说“按我的风格写”“像我以前文章那样写”
- 用户要把零散素材整理成可发布公众号文
- 用户提供控制论、架构、Agent、测试、工程经验类素材
- 用户要求“既有观点也有落地方法”

## 输入最小要求
至少拿到以下两项：

1. 文章主题
2. 目标读者（如：后端工程师、AI 工程师、泛技术读者）

若缺失，先补问再写。

## 输出默认结构
除非用户指定其他结构，默认采用：

1. 前言 / 事件起点
2. 基础概念（用生活类比降维解释）
3. 映射到工程问题
4. 失效模式与成因
5. 可执行治理策略
6. 方法升级（从技巧到体系）
7. 结语（一句话收束）

## 写作方法（执行步骤）

### Step 1: 先抽“主线句”
先用一句话定义全文主线：

- 模板：`我想说明：<一个底层规律>，它如何解释 <当前工程问题>，并指导 <可执行动作>`

后续每节都要服务这条主线，避免散。

### Step 2: 先故事，后术语
每个关键概念先给生活画面，再补技术名词。

- 例：先写“洗澡调水温”，再说“闭环反馈”
- 例：先写“骑车左右猛打把”，再说“振荡/增益过高”

### Step 3: 做一一映射
遇到抽象概念，必须给出映射表或映射段：

- 目标 ↔ 用户需求
- 传感器 ↔ 日志/测试反馈
- 控制器 ↔ 模型决策
- 执行器 ↔ 工具调用
- 被控对象 ↔ 代码库/运行环境

### Step 4: 给判断标准，不只给观点
每个“问题”都尽量附一个可量化判断方式。

例如：
- 收敛：未通过用例数单调下降
- 振荡：A/B 方案来回切换
- 发散：每轮新增问题数持续上升

### Step 5: 给动作清单
观点段后面紧跟动作项（列表/表格），让读者可立即执行。

### Step 6: 保留边界意识
避免绝对化口吻。对关键建议补一句边界说明：

- 适用场景
- 不适用场景
- 成本与风险

## 风格硬约束

1. 用第一人称，但避免“自我表演”。
2. 技术术语首次出现要有一句人话解释。
3. 长短句交替，避免整段同节奏。
4. 少用口号，多用机制解释（为什么会这样）。
5. 不堆空洞形容词，尽量落在可观察现象。

## 内容组织建议

- 复杂对照优先表格。
- 操作步骤优先编号列表。
- 关键抽象关系可用简图（若用户未要求，不强制）。
- 代码片段后必须解释“这段代码说明了什么”。
